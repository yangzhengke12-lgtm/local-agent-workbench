# AI-Agent 管理系统：核心定位与 v4 迭代方案

> 目的：给后续实现者阅读，明确系统从“AI 管理团队”升级为“Agentic Workflow Runtime”的方向。

---

## 1. 之前的核心定位

之前系统的核心定位更像是：

> 一个拟组织化的 AI 团队管理系统，用 Manager 来分配任务、组织讨论、评价员工、沉淀知识。

其结构大致是：

```text
用户
↓
Manager 正经理
↓
Worker 员工
↓
Deputy 副经理复核
↓
任务看板 / 知识库 / 评分 / 圆桌讨论
```

也就是说，旧系统的中心是 **Manager 调度 Agent**，不是 Runtime 执行工作流。

旧系统主要能力：

- 角色协作：Alex 写代码，Sophia 审查，Marcus 运维，Elena 写文档，Nathaniel 测试，Victor 复核。
- Manager 决策：根据用户请求判断该找谁、是否圆桌讨论、是否请副经理。
- 任务看板：用 `task_board.json` 管理 `todo / in_progress / done / failed`。
- 知识库：用 `team_knowledge.json` 沉淀经验、模板、反模式。
- 评分体系：用 `worker_scores.json` 记录 Worker 质量表现。
- 项目分工：`project_setup()` 可以把项目拆成 pipeline，但更像“生成分工表”，不是严格执行引擎。
- 会话记忆：Worker 有上下文记忆，方便多轮协作。

旧系统的问题不是方向错，而是内核还停留在“管理隐喻”：

```text
经理安排员工干活
```

而新的目标应该是：

```text
工程运行时调度可验证任务节点
```

---

## 2. 新的核心定位

新的核心定位应该是：

> 一个面向软件工程任务的 Agentic Workflow Runtime。

聊天可以保留，但聊天只是控制入口，不是系统内核。

新结构应该是：

```text
User / Chat
↓
Control Plane 人类控制面
↓
Workflow Runtime 状态机 / DAG / 重试 / 暂停 / 恢复
↓
Agent Layer Manager / Worker / Verifier / Deputy
↓
Tool + Workspace Layer 文件、命令、Git、测试、文档
```

旧系统主语是：

```text
Agent / Worker
```

新系统主语应该是：

```text
Workflow / TaskNode / Runtime
```

旧系统关心：

```text
这个任务该交给哪个员工？
员工完成得怎么样？
要不要副经理复核？
```

新系统应该关心：

```text
这个节点是否 ready？
它的输入输出合约是什么？
产物是否通过 verifier？
失败是否可重试？
下游是否应该阻断？
状态能否恢复？
预算是否超限？
```

因此不是推翻旧系统，而是重新分层：

- Manager / Worker / Deputy 变成 **Agent Layer**。
- 任务看板变成 **Workflow 状态视图**。
- 评分体系变成 **Verifier / Quality Metrics**。
- 知识库变成 **经验缓存 / Policy Memory**。
- 圆桌讨论变成 **Planning / Review 策略之一**。
- 聊天界面变成 **Control Plane**。

一句话：

> 之前是 AI 公司组织模拟；现在要升级成 Agentic Workflow Runtime。

---

## 3. 可参考项目与借鉴点

v4 不建议对标纯多 Agent 聊天框架，而应该参考工程运行时、coding agent、DAG 编排和隔离执行相关项目。

推荐参考：

- LangGraph：学习 durable execution、状态机、checkpoint、human-in-the-loop。
- Open SWE：学习 sandbox、coding agent、subagent、验证和 PR 流程。
- open-multi-agent：学习 goal → task DAG → 并行执行 → 汇总结果。
- Phasr：学习多 agent workspace、git worktree 隔离、review-first merge。
- Orloj：学习 agent-as-code、DAG、policy、retry、token cap、observability。
- CrewAI / AutoGen：可以参考 Agent / Task 抽象，但不要让其角色扮演式设计决定内核。

重点原则：

> 把 LLM 从“流程控制者”降级为“智能函数”。流程控制必须由代码掌握。

LLM 适合负责：

- 分析需求。
- 生成计划候选。
- 编写代码。
- 审查代码。
- 解释失败。
- 提出修复方案。

代码必须负责：

- 节点能否进入下一步。
- 是否重试。
- 是否阻断下游。
- 是否超预算。
- 是否需要人工确认。
- 哪些产物被采纳。

## 4. v4 核心目标

v4 的目标不是继续增加角色感，而是增加执行语义。

核心目标：

> 把当前“Manager 手动调度 + Worker 自由文本汇报”升级为“状态机驱动、结构化交付、自动验证、可重试、可恢复的工程工作流”。

v4 验收标准：

```text
给系统一个项目目标
→ 自动拆成 DAG
→ 只执行 ready 节点
→ Worker 交付结构化结果
→ Verifier 验证
→ 不通过自动重试
→ 下游依赖正确阻断
→ 中断后能从 JSON 状态继续
→ 最终能输出可审计 run report
```

---

## 5. v4 Runtime Kernel

v4 应先实现最小运行时内核，不急着做 GitHub PR、复杂 UI 或更多角色。

建议新增核心模型：

```text
WorkflowRun
TaskNode
WorkerResult
VerificationResult
Artifact
Budget
```

### 5.1 TaskNode 状态机

每个任务节点使用固定状态机：

```text
todo → ready → running → verifying → done
                         ↘ retrying
                         ↘ failed
                         ↘ blocked
                         ↘ needs_replan
```

每个节点至少包含：

```text
id
name
description
depends_on
assigned_worker
status
attempts
max_retries
artifacts
verification
budget
error
created_at
updated_at
```

重点不是“图很酷”，而是：

- 状态可恢复。
- 节点可重跑。
- 流程可暂停。
- 失败可审计。

---

## 6. v4 结构化合约

结构化合约是整个系统的地基。

Worker 不再只返回自由文本，必须归一化成 `WorkerResult`。

### 6.1 WorkerResult

建议结构：

```json
{
  "status": "success|partial|failed|needs_review",
  "summary": "...",
  "artifacts": [],
  "issues": [],
  "needs_replan": false,
  "retryable": true,
  "confidence": 0.8,
  "raw_text": "..."
}
```

### 6.2 VerificationResult

Verifier 返回：

```json
{
  "verdict": "pass|reject|needs_retry|needs_replan",
  "score": 4,
  "blocking_issues": [],
  "retry_instruction": "...",
  "raw_text": "..."
}
```

### 6.3 关键原则

> 结构化是目标，兜底解析是底线。

LLM 不输出 JSON 时，系统不能崩。

兜底策略：

- Worker 自由文本包装成 `status = needs_review`。
- Verifier 自由文本包装成 `verdict = needs_retry`。
- 原始内容保留在 `raw_text`。

## 7. v4 验证闭环

新增 `delegate_with_verification`，形成第一条真正工程闭环：

```text
run_worker
→ normalize WorkerResult
→ Sophia review
→ Nathaniel validation
→ merge VerificationResult
→ pass / retry / reject / replan
```

默认策略：

- `pass`：节点完成。
- `needs_retry`：把 verifier 的反馈拼入下一轮任务。
- `reject`：可重试则重试，不可重试则 failed。
- `needs_replan`：停止当前节点，交给 runtime 重规划。

注意：

- 不要只让 agent “说完成了”。
- 关键节点要尽量有验证动作，例如测试、lint、读文件、检查产物。
- `evaluate_result` 可以继续保留评分，但不能只是记分，应该能产出 verdict。

## 8. v4 DAG Pipeline Engine

当前 `project_setup()` 会生成 pipeline，但 `depends_on` 没有真正被强制执行。

v4 需要新增：

```text
run_project_pipeline(project_name)
```

行为：

- 读取 `project_states/{project_name}_state.json`。
- 找出依赖已完成的 `todo` 节点。
- 并行执行互不依赖的节点。
- 失败节点阻断下游。
- 每个节点都走 `delegate_with_verification`。
- 每轮保存状态，支持中断后继续。

节点状态更新规则：

- 验证通过：`done`。
- 验证不通过且可重试：`retrying`，随后再次执行。
- 超过重试次数：`failed`。
- 依赖失败：`blocked`。
- 计划错误：`needs_replan`。

---

## 9. v4 收敛模式

新增：

```text
run_convergence_loop(task, worker, verifier, stable_rounds=2, max_rounds=5)
```

适用任务：

- 持续找 bug。
- 持续审查代码。
- 持续补测试。
- 持续改文档直到无新增问题。

停止条件：

```text
连续 N 轮无新增 issue
或达到 max_rounds
或预算耗尽
或 verifier 判定 pass
```

这个能力比“让 Sophia 再看看”更工程化，因为它有明确停止条件。

---

## 10. v4 Workspace 隔离 Lite

v4 不一定立刻上完整 sandbox，但要先留出隔离接口。

建议接口：

```text
WorkspaceProvider
- current
- temp_copy
- git_worktree
```

默认可以继续使用 current workspace。

高风险或并行写文件任务，应逐步支持 `git_worktree`。

原因：

> 多个 agent 并行改代码，如果没有隔离，迟早互相踩文件。

---

## 11. v4 Budget + Policy

v4 应加入最小预算系统。

每个 workflow / node 至少有：

```text
max_attempts
max_rounds
max_tool_calls
max_runtime_seconds
max_model_calls
```

先不做精确 token 统计也可以，但必须有轮次和调用次数熔断。

默认 Policy：

- 写文件任务必须验证。
- 高风险命令需人工确认。
- 连续失败 2 次触发 Deputy。
- `needs_replan` 不自动继续执行，先生成 replan proposal。

---

## 12. Human Control Plane

不要急着做复杂 UI。

先让控制面具备这些命令/能力：

```text
status
pause
resume
approve
reject
replan
retry_node
show_artifacts
show_trace
```

人类不是一直聊天，而是在关键节点审批、修改状态、恢复流程。

---

## 13. v4 推荐优先级

建议优先级：

1. P0：`TaskNode` 状态机。
2. P0：`WorkerResult` / `VerificationResult` 结构化合约。
3. P0：`delegate_with_verification` 验证闭环。
4. P1：`run_project_pipeline` DAG 执行。
5. P1：`run_convergence_loop` 收敛模式。
6. P1：`WorkflowRun` 持久化和 resume。
7. P2：`WorkspaceProvider` 隔离接口。
8. P2：`Budget` / `Policy`。
9. P2：Human approval gates。
10. P3：GitHub Issue / PR 集成。
11. P3：Web Dashboard。

---

## 14. 与原 v4 方案的区别

原 v4 方案偏“给现有系统加能力”：

```text
Manager 工具增强
结构化输出
验证重试
Pipeline 执行
```

现在的 v4 方案是“内核升级”：

```text
Runtime Kernel
TaskNode 状态机
WorkflowRun 持久化
Verifier 合约
Workspace 隔离接口
Budget / Policy
Human Control Plane
```

结论：

> v4 不要先做 GitHub PR，也不要先做漂亮 UI。先把 Runtime Kernel 做扎实。

## 15. 给实现者的执行建议

如果当前代码仍以 `manager.py` 为主，建议先单文件演进，不要一开始大拆包。

推荐落地顺序：

1. 增加 schema 常量和 normalize 函数。
2. 让 `run_worker` 返回 `structured_result`，但保留现有返回结构。
3. 新增 `verify_worker_result`。
4. 新增 `delegate_with_verification`。
5. 改造 `evaluate_result`，增加 verdict 字段。
6. 新增 `run_project_pipeline`。
7. 新增 `run_convergence_loop`。
8. 新增轻量 `request_replan`。
9. 接入 Manager tools。
10. 增加最小测试，优先覆盖纯函数和状态流转。

注意：

- 不要破坏现有 `delegate_task`。
- 不要引入数据库。
- 不要假设 LLM 一定输出合法 JSON。
- 不要让自然语言成为流程判断依据。
- 每一步执行结果都要能持久化、审计、恢复。
